# Skill: Forms And Validation

## Purpose

Use this skill when implementing forms, validation, default values, form-to-payload mapping, and multi-section workflows.

## Project Evidence

- `src/components/organisms/AddOrEditJobForm/AddOrEditJobForm.tsx`: composed form with default values, validation context, submit mapping, and feature sections.
- `src/components/organisms/AddOrEditJobForm/schema.ts`: schema is isolated from JSX.
- `src/components/organisms/AddOrEditJobForm/jobFormUtils.ts`: default values and API payload mapping are pure functions.
- `src/components/molecules/AssignedAssetsFormPart/schema.ts`: nested field validation is colocated with the form section.
- `src/components/molecules/CrewAndPumperSection/schema.ts`: conditional validation is isolated and typed.
- `tests/components/organisms/AddOrEditJobForm/jobFormUtils.spec.ts`, `tests/components/organisms/AddOrEditJobForm/schema.spec.ts`: validation and mapping logic are tested separately.

## Current Pattern

Forms are composed from reusable sections. Validation schemas, default-value builders, and payload mappers are separated from rendering. Complex field arrays, conditional validation, and domain-specific submit behavior are tested as pure logic where possible.

## Implementation Guidelines

- Keep schema/validation rules outside JSX.
- Build default values with named functions.
- Map form values to API payloads in utilities, not in submit JSX.
- Split large forms into smaller field-section components.
- Keep conditional validation close to the section that owns the condition.
- Disable or guard submit actions based on validity, dirty state, pending state, and required data availability.
- Test schemas and payload mappers when they contain business rules.

## Recommended Structure

```text
FeatureForm/
  FeatureForm.tsx
  schema.ts
  formUtils.ts
  types.ts

FeatureFormSection/
  FeatureFormSection.tsx
  schema.ts
```

## Good Example

```ts
export const mapFormValuesToPayload = (values: FormValues): CreateItemPayload => ({
  name: values.name.trim(),
  relatedIds: values.relatedItems.flatMap(item => item.ids),
  optionalNote: values.note || null,
});
```

This follows the project pattern of keeping submit payload construction in pure utility functions.

## Anti-patterns

- Building API payloads inline inside `onSubmit`.
- Duplicating validation rules across components.
- Keeping multi-section form logic in one large component.
- Allowing nullable fields to pass through without explicit conversion.
- Leaving important validation untested.

## Checklist

- Schema is isolated.
- Default values are generated by named functions.
- Form-to-payload mapping is pure and testable.
- Large forms are split into sections.
- Conditional validation is explicit.
- Complex validation and mapping have tests.
